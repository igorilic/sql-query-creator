# SQL Query Creator — Requirements

**Source:** `spec.md` (project root)
**Date:** 2026-03-20
**Pipeline:** github-feature (feature-20260320-120720)

---

## 1. Problem Statement

Developers frequently write SQL queries manually, which is slow for complex queries and error-prone when unfamiliar with a database's schema. This tool lets developers describe queries in plain English and receive syntactically correct, schema-aware SQL generated by a local LLM (LM Studio). By introspecting the connected database, the tool provides the LLM with full schema context, producing accurate queries without sending data to external services.

## 2. User Stories

| ID    | Story | Priority |
|-------|-------|----------|
| US-01 | As a developer, I want to connect to a PostgreSQL or SQLite database so the app can introspect its schema. | Must |
| US-02 | As a developer, I want to browse the database schema (tables, columns, types) in a sidebar so I understand the data model at a glance. | Must |
| US-03 | As a developer, I want to describe a query in plain English and receive generated SQL so I can prototype queries faster. | Must |
| US-04 | As a developer, I want to have a conversational follow-up (e.g., "now add a JOIN") so the LLM refines the previous query incrementally. | Must |
| US-05 | As a developer, I want to view the generated SQL in an editable, syntax-highlighted editor so I can tweak the output. | Must |
| US-06 | As a developer, I want to copy the generated SQL to my clipboard with one click. | Must |
| US-07 | As a developer, I want the app to work without a database connection so I can ask generic SQL questions. | Should |
| US-08 | As a developer, I want to see the connection status and current database type in the header so I know the app's state. | Must |

## 3. Functional Requirements

### FR-01: Database Connection Management
- FR-01.1: The user MUST select a database type (PostgreSQL or SQLite) before providing credentials.
- FR-01.2: For PostgreSQL, the user provides host, port, database name, username, and password.
- FR-01.3: For SQLite, the user provides a file path to the `.db` / `.sqlite` file.
- FR-01.4: Only one active connection is allowed at a time; establishing a new connection replaces the previous one.
- FR-01.5: On successful connection, the app introspects the schema automatically (tables, columns, data types, foreign key relationships).
- FR-01.6: Connection errors are displayed to the user with actionable messages.

### FR-02: Schema Browser
- FR-02.1: A sidebar or collapsible panel displays all tables in the connected database.
- FR-02.2: Each table is expandable to show its columns with data types.
- FR-02.3: The schema browser updates when a new connection is established.
- FR-02.4: When no database is connected, the schema browser shows an empty/placeholder state.

### FR-03: Natural Language Chat Interface
- FR-03.1: A text input field accepts plain English query descriptions.
- FR-03.2: The app sends the user's message, conversation history, and (if connected) the database schema to the LLM.
- FR-03.3: Conversation history is maintained within a session for follow-up refinements.
- FR-03.4: The LLM response containing SQL is parsed and rendered in the query editor.
- FR-03.5: Non-SQL explanatory text from the LLM is displayed in the chat panel.
- FR-03.6: The chat works without a database connection; schema context is simply omitted from the prompt.

### FR-04: Query Editor
- FR-04.1: Generated SQL is displayed in a code editor with SQL syntax highlighting.
- FR-04.2: The editor content is editable by the user.
- FR-04.3: A "Copy" button copies the current editor content to the system clipboard.
- FR-04.4: The editor updates when a new query is generated from the chat.

### FR-05: LM Studio Integration
- FR-05.1: The app communicates with LM Studio via the OpenAI-compatible API at a configurable base URL (default: `http://localhost:1234/v1`).
- FR-05.2: The app uses the `/chat/completions` endpoint.
- FR-05.3: The app uses whichever model is currently loaded in LM Studio (no model selection in the UI).
- FR-05.4: If LM Studio is unreachable, the app displays a clear error message.

### FR-06: Offline / No-Connection Mode
- FR-06.1: The app loads and is usable without a database connection.
- FR-06.2: Users can ask generic SQL questions and receive example queries.
- FR-06.3: Schema context is omitted from LLM prompts when no database is connected.

## 4. Non-Functional Requirements

| ID     | Requirement | Target |
|--------|-------------|--------|
| NFR-01 | **Performance**: Schema introspection completes within 5 seconds for databases with up to 200 tables. | ≤ 5s |
| NFR-02 | **Performance**: LLM response streaming should begin rendering within 2 seconds of sending the request (network dependent on LM Studio). | ≤ 2s first token |
| NFR-03 | **Security**: Database credentials are never persisted to disk or sent to the LLM. They are held in memory only for the duration of the session. | — |
| NFR-04 | **Security**: The app runs entirely locally; no data leaves the user's machine (LM Studio is also local). | — |
| NFR-05 | **Accessibility**: UI is keyboard-navigable and uses semantic HTML. | WCAG 2.1 AA |
| NFR-06 | **Extensibility**: The architecture accommodates future query execution and result display without major refactoring. | — |
| NFR-07 | **Browser Support**: Modern evergreen browsers (Chrome, Firefox, Safari, Edge — latest 2 versions). | — |

## 5. Acceptance Criteria

### AC-01: Database Connection (US-01, FR-01)
- **Given** the user is on the connection screen,
  **When** they select "PostgreSQL" and enter valid credentials,
  **Then** a connection is established and the header shows "Connected — PostgreSQL".
- **Given** the user is on the connection screen,
  **When** they select "SQLite" and provide a valid file path,
  **Then** a connection is established and the header shows "Connected — SQLite".
- **Given** the user provides invalid credentials,
  **When** they attempt to connect,
  **Then** an error message is displayed describing the failure reason.
- **Given** an active PostgreSQL connection exists,
  **When** the user connects to a new SQLite database,
  **Then** the previous connection is closed and the schema browser updates to the new database.

### AC-02: Schema Browser (US-02, FR-02)
- **Given** a database connection is active,
  **When** the schema browser loads,
  **Then** all tables are listed in the sidebar.
- **Given** a table is displayed in the schema browser,
  **When** the user expands it,
  **Then** its columns and data types are visible.
- **Given** no database is connected,
  **When** the user views the schema browser,
  **Then** a placeholder message (e.g., "Connect a database to browse schema") is shown.

### AC-03: Natural Language Query Generation (US-03, FR-03)
- **Given** a database is connected with schema loaded,
  **When** the user types "get all users who signed up last month" and submits,
  **Then** the LLM returns SQL that references the actual table/column names from the schema, and it is displayed in the query editor.
- **Given** LM Studio is not running,
  **When** the user submits a query,
  **Then** an error message indicates the LLM service is unreachable.

### AC-04: Conversational Follow-ups (US-04, FR-03)
- **Given** a SQL query was just generated,
  **When** the user types "now add a JOIN to the orders table",
  **Then** the LLM produces an updated query incorporating the JOIN, and the query editor updates.
- **Given** a conversation with 3+ messages,
  **When** the user sends a follow-up,
  **Then** the full conversation history is included in the LLM context.

### AC-05: Query Editor (US-05, US-06, FR-04)
- **Given** SQL has been generated,
  **When** the user views the query editor,
  **Then** the SQL is syntax-highlighted.
- **Given** SQL is displayed in the editor,
  **When** the user manually edits the text,
  **Then** the changes are preserved in the editor.
- **Given** SQL is in the editor,
  **When** the user clicks "Copy",
  **Then** the editor content is copied to the system clipboard.

### AC-06: Offline Mode (US-07, FR-06)
- **Given** no database is connected,
  **When** the user asks "how do I query date ranges in PostgreSQL?",
  **Then** the LLM responds with example SQL and the response appears in the chat.

### AC-07: LM Studio Integration (FR-05)
- **Given** LM Studio is running at the configured URL,
  **When** the app sends a chat completion request,
  **Then** a valid response is received and parsed.
- **Given** LM Studio is not running,
  **When** the app attempts to communicate,
  **Then** a user-friendly error is shown (not a raw network error).

## 6. Constraints

- **C-01**: Frontend must use Next.js with TypeScript.
- **C-02**: UI components must use the Catalyst UI Kit (Tailwind CSS), structured as a shared package in the monorepo.
- **C-03**: Monorepo layout: `apps/web/` (Next.js app), `packages/ui/` (Catalyst UI Kit).
- **C-04**: LLM communication is exclusively through LM Studio's local OpenAI-compatible API.
- **C-05**: Only PostgreSQL and SQLite are supported in this scope.
- **C-06**: Database connections are made from Next.js API routes (server-side) — credentials must not be exposed to the browser.

---

## 7. Out of Scope

- Query execution and result display (future extension)
- Query history / saved queries
- Multiple simultaneous database connections
- Model selection within the app (delegated to LM Studio)
- Support for databases other than PostgreSQL and SQLite
- User authentication or multi-user support
- Schema diff / migration suggestions
- Deployment to cloud (this is a local-first tool)

## 8. Open Questions

| # | Question | Impact |
|---|----------|--------|
| OQ-1 | Should the LM Studio base URL be configurable from the UI, or only via environment variable? | Affects settings UI scope |
| OQ-2 | Should the app support streaming responses from LM Studio (SSE), or wait for the full response? Streaming would improve UX significantly. | Affects API route design and chat rendering |
| OQ-3 | What is the maximum conversation history length before truncation? Long conversations may exceed LLM context window limits. | Affects prompt engineering and memory management |
| OQ-4 | Should foreign key relationships be visualized in the schema browser (e.g., arrows, badges), or just listed as column metadata? | Affects schema browser UI complexity |
| OQ-5 | Should the system prompt to the LLM be customizable by the user, or hardcoded? | Affects settings scope |
| OQ-6 | How should the Catalyst UI Kit be integrated — as a copy of source files into `packages/ui/`, or as an npm dependency? The existing `catalyst-ui-kit/` directory in the repo suggests a source copy approach. | Affects monorepo setup |

## 9. Assumptions

| # | Assumption | Rationale |
|---|-----------|-----------|
| A-1 | LM Studio is already installed and running on the user's machine before using the app. The app does not manage LM Studio's lifecycle. | Spec describes LM Studio as a prerequisite, not a managed dependency. |
| A-2 | Database connection logic runs server-side via Next.js API routes to avoid exposing credentials to the browser. | Standard security practice; spec doesn't specify but this is the only safe approach. |
| A-3 | The Catalyst UI Kit will be copied/vendored into `packages/ui/` from the existing `catalyst-ui-kit/` directory in the repo. | The `catalyst-ui-kit/` directory with demo apps already exists in the project root. |
| A-4 | The app uses npm/pnpm workspaces for the monorepo structure. | Spec defines a monorepo with `apps/` and `packages/`; workspaces are the standard approach for Next.js monorepos. |
| A-5 | LLM response streaming (SSE) will be implemented for better UX, as the OpenAI-compatible API supports it natively. | Industry standard for chat UIs; would be poor UX to wait for full response. |
| A-6 | The conversation is scoped to a single session (browser tab). Refreshing the page clears the chat history. | No persistence requirement in spec; query history is explicitly out of scope. |
| A-7 | Schema introspection includes tables, columns, data types, and foreign keys but not views, stored procedures, or indexes. | Spec mentions "tables, columns, types, relationships" — interpreting "relationships" as foreign keys only. |
| A-8 | The SQL generation prompt includes the database dialect (PostgreSQL vs SQLite) so the LLM generates dialect-appropriate SQL. | Necessary for correct SQL generation; implicit in the spec. |

|----------|--------|

| Full flow: connect to SQLite → introspect schema → schema appears in sidebar | AC-01, AC-02 |

### Edge Cases & Error Scenarios
| Scenario | Expected Behavior |
|----------|-------------------|
| LM Studio is not running when app starts | App loads; error shown only when user submits a query |
| Database connection drops mid-session | Graceful error; schema browser shows stale data or reverts to placeholder |
| Empty database (no tables) | Schema browser shows "No tables found" message |
| LLM returns malformed/no SQL | Chat displays the raw response; editor is not updated with garbage |
| Very large schema (200+ tables) | Introspection completes within NFR-01 target; schema is paginated or virtualized in sidebar |
| User submits empty message | Input is not sent; button is disabled or no-op |
| Concurrent rapid message sends | Messages are queued or second send is disabled while LLM is responding |
